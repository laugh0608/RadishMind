# RadishMind 代码规范

更新时间：2026-05-06

## 文档目的

本文档定义 RadishMind 仓库的长期代码风格与实现约束。它不替代语言官方规范，也不追求覆盖所有细枝末节；它只固定本仓库最容易失控的工程边界：命名、抽象、协议、脚本组织和验证。

当前实现按职责分层：模型训练、评测与脚本优先 `Python`，前端 UI 默认 `React + Vite + TypeScript`，服务 / `gateway` / `API` 可采用 `Go`。后续若引入其它语言，应先遵循对应语言生态的惯用实践，再按本文档保持职责清晰、协议显式和实现可验证。

## 总原则

- 代码应趋近对应语言的良好实践，优先清晰、直接、可测试和可维护。
- 实现应服务于当前阶段目标，不提前把未来复杂度压进当前阶段。
- 优先修正根因，不用层层 fallback 掩盖协议、数据或职责边界错误。
- 能规则化、schema 化或工具化的逻辑，不强行交给模型自由生成。
- 能用标准库、明确类型和小函数表达的问题，不写成动态魔法或多层转发。

## 命名规范

- 模块、函数、变量和类型名必须表达真实职责、领域对象或数据形态。
- 函数名应说明动作和对象，例如 `build_copilot_request`、`validate_gateway_envelope`。
- 布尔变量和返回值应表达判断语义，例如 `requires_confirmation`、`is_schema_valid`。
- 文件名应与主要职责一致，不把多个无关职责塞进同一文件。
- 避免泛名：`process_data`、`handle_item`、`do_work`、`manager`、`helper`、`utils`。
- 若只能想到泛名，通常说明职责边界还没有想清楚，应先收窄问题。

## 抽象规范

只有满足以下至少一项时，才引入 helper、builder、adapter、service 或其它抽象：

- 抽象承载稳定职责边界，例如 request builder、response builder、schema validator。
- 抽象消除了真实重复，而不是为了少写几行代码。
- 抽象让错误处理、测试或协议边界更清楚。
- 抽象能降低调用侧认知成本，而不是隐藏简单逻辑。

禁止以下抽象：

- 只转发参数、不增加语义的空转 wrapper。
- 没有清晰领域职责的万能 manager / factory / helper。
- 多层嵌套转发后才到真实逻辑的调用链。
- 为了“以后可能通用”提前做的过度泛化。
- 用隐式 fallback 链吞掉 schema、任务或数据错误。

## Python 规范

- 优先使用标准库和仓库内既有 helper，不为常见任务引入不必要依赖。
- 函数保持单一职责；当函数开始同时处理读取、转换、校验、写入和报告时，应评估拆分。
- 对结构化数据优先使用 `dict` / `list` 的明确 schema 约束、`dataclass` 或清晰类型别名，不依赖散落字符串约定。
- 错误应显式暴露为异常、返回状态或 audit 记录，不要静默吞掉。
- 路径处理优先使用 `pathlib.Path`，避免手拼分隔符。
- JSON 读写必须保持 UTF-8、稳定缩进和可复验字段。
- 避免过度元编程、动态 import、隐式全局状态和运行时猴子补丁。
- 需要排序、归一化或 canonicalization 时，应把规则写成可测试的确定性函数。

## TypeScript / React 规范

- UI 优先函数组件和明确类型，不把业务协议散落成多个重复的本地状态副本。
- 副作用、数据获取和状态转换应分层处理，不把复杂逻辑挤进组件渲染过程。
- 组件命名与目录结构应能直接表达页面、容器或业务单元职责。
- 不默认堆叠 `useMemo` / `useCallback`，只在真实性能或引用稳定性需要时使用。
- UI 只消费后端协议与视图模型，不把业务真相源复制到前端。

## Go 规范

- `Go` 优先承载服务、`gateway`、HTTP API、长驻进程和并发边界清晰的基础设施层。
- 优先标准库和轻依赖，包边界要小而明确，避免把所有逻辑塞进单一 `main` 或万能 service 包。
- 错误、超时、取消和并发控制应显式处理，`context.Context` 必须贯穿请求生命周期。
- HTTP handler、配置解析、鉴权、重试和序列化逻辑应保持可测试、可替换、可审计。
- `Go` 层同样不应直接写上层业务真相源，只负责平台协议、路由和受控执行。

## 协议与数据

- 统一协议优先采用结构化 JSON 和 JSON Schema。
- 新增字段必须明确语义、来源、是否可选、是否可由模型生成、是否可由工具层生成。
- `status`、`risk_level`、`requires_confirmation`、citation、action shape 等硬字段优先由规则层或 builder 收口。
- 长语义、批次说明和样本描述应进入 manifest、record、fixture 或 metadata，不编码进长目录名和长文件名。
- committed JSON 文件接近体量上限时，优先拆成主索引和 `.parts/` 分片。

## 脚本组织

- `scripts/` 根目录只放稳定入口、跨平台包装脚本和少量高频直达命令。
- 内部实现、检查逻辑、fixture helper 和较长实现应放入浅层分类目录，例如 `scripts/checks/`、`scripts/eval/`。
- 新脚本必须有清晰输入、输出和失败语义；不要依赖调用者猜测副作用。
- 会写入 committed 资产的脚本，应支持 check / dry-run 或 summary 校验路径。
- 长时间运行、加载本地模型、下载数据或显著占用资源的脚本，不作为默认自动验证入口。

## 文档与注释

- 注释用于解释非显而易见的约束、领域规则或失败处理，不重复代码表面动作。
- 文档应说明当前阶段、当前结论、下一步和停止线，不堆历史流水。
- 入口文档保持短；长实验观察、批次细节和命令输出进入周志、manifest、summary 或 run record。

## 验证要求

- 修改后优先执行当前环境的仓库级验证入口：Linux / WSL 用 `./scripts/check-repo.sh`，Windows / PowerShell 用 `pwsh ./scripts/check-repo.ps1`。
- 文档改动至少执行文本卫生和仓库级基线；若没有执行，应在总结中说明原因。
- 代码改动应按风险补充或更新针对性检查，不只依赖人工阅读。
- 若新增抽象，验证应覆盖抽象的输入、输出、失败语义和调用侧不变量。

## 判断标准

一个实现通常可以接受，当它同时满足：

- 读者能从命名和文件位置理解职责。
- 数据边界、协议边界和错误边界是显式的。
- 抽象减少了真实复杂度，而不是增加绕路。
- 现有文档、schema、任务卡和验证入口保持一致。
- 后续维护者不需要依赖聊天历史才能判断为什么这样写。
