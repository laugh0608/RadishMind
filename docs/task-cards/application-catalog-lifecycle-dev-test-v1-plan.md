# 用户工作区应用目录与生命周期（开发/测试态）v1 实施任务卡

更新时间：2026-07-13

状态：`application_catalog_lifecycle_dev_test_v1_complete`

## 任务目标

按照[用户工作区应用目录与生命周期（开发/测试态）v1](../features/user-workspace/application-catalog-lifecycle-dev-test-v1.md)，把现有预置只读应用摘要升级为显式启用、可持久化、可审查的开发测试态应用目录，并连续交给配置草案、API 接入、Gateway 调试台、请求历史和发布审查。

## 前置条件

- 用户工作区现有应用列表、配置草案、API 接入、Gateway 调试台、请求历史和发布审查已经完成开发测试态闭环。
- `applications:read` 已存在；本任务新增并分离 `applications:write` 与 `applications:archive`。
- 当前成熟度仍为内部开发者预览；正式成员关系、生产应用存储库、API 密钥、配额、计费和正式晋级不在本批次成立。

## 实现范围

1. 建立 `application_catalog_record.v1`、服务端应用标识、元数据校验、所有者作用域、生命周期、版本和稳定失败语义。
2. 建立 `ApplicationCatalogRepository`，实现 `memory_dev` 与独立 `postgres_dev_test`，两种模式共享精确读取、排序、筛选、CAS 和软归档语义。
3. 显式目录模式接管 `GET /v1/user-workspace/applications`，并新增创建、详情、完整元数据更新和归档端点；目录存储失败不回退预置列表或内存。
4. 建立独立 schema、迁移清单、校验和、迁移运行器、存储选择器和破坏性 PostgreSQL 集成验证。
5. 更新 Web 应用页面，支持活跃 / 已归档筛选、创建、详情、编辑冲突、归档确认和应用上下文切换。
6. 配置草案保存、发布候选创建与审查消费精确活跃应用检查；既有历史保持只读。
7. 完成真实浏览器连续路径、重启恢复、敏感信息检查、当前焦点和周志收口。

## 高风险边界

- 应用目录是新增 API、schema 和写入边界，必须拒绝未知字段、敏感材料和客户端伪造的身份 / 版本 / 审计字段。
- 应用目录模式只能有一个真相源；目录存储不可用时必须显式失败。
- 更新与归档必须执行原子 CAS，归档只允许 `active -> archived`，不提供反归档或物理删除。
- OIDC 集成测试模式在成员关系未成立前必须零查询返回 `workspace_membership_unavailable`。
- Web 阻断不解释为 Gateway 原始上行 API 的生产授权。

## 验收证据

- Go：领域校验、标识、所有者作用域、权限分离、列表筛选 / 游标、CAS、归档和零查询负向测试。
- PostgreSQL：迁移应用 / 重复应用 / 回滚 / 重应用、运行角色 DDL 拒绝、并发单一成功、重启恢复和无回退。
- Web：离线零请求、严格响应校验、创建 / 更新 / 归档、冲突保留、应用切换清理和归档后交接阻断。
- 浏览器：空目录创建到配置、调用、历史、候选、多标签冲突、归档、只读历史和重启恢复。
- 门禁：Go 单测 / 竞态 / 静态检查、Web 单测 / 构建、PostgreSQL 集成、`git diff --check`、快速与完整仓库检查。

## 完成条件

- 用户可在当前工作区创建、管理和归档自己的开发测试态应用，版本与审计链可解释、可复验。
- 目录模式下应用身份、元数据、生命周期和所有者只有一个真相源；失败不回退。
- 归档后禁止新的配置、发布审查和 Web 调用交接，但既有草案、候选和请求历史保持可读。
- 正式成员关系、生产授权、生产密钥、配额、计费、反归档、删除和正式晋级继续保持停止线。
