## 关键决策

- `suggest_flowsheet_edits` 的首批真实 capture 继续沿现有 `run-radishflow-suggest-edits-poc-batch.py` 入口推进，不另起第二套临时脚本
- 当前不把 `deepseek` 首条坏输出直接入仓为正式正向批次；只有通过当前回归 / audit 门槛的真实输出才进入正式 `candidate-records`
- 当前先把 provider 失败面区分为三类：可用性阻塞、额度阻塞、任务质量漂移；后续治理按这三个层次分别推进，而不是统一归因成“模型答坏了”
- Claude 路由继续保留 Anthropic `messages` API 风格，与其余走 OpenAI-compatible 的 profile 分开处理；当前 profile fallback 只在 provider 不可用或显式失败时切换，不在“返回成功但质量一般”时自动换档
- `suggest_flowsheet_edits` 当前优先依赖“窄范围 canonicalization + teacher provider 首样本回归”推进，而不是继续为同一任务叠加更宽泛的 prompt 兜底
- 当前把 `2026-04-13-radishflow-suggest-edits-poc-real-v3` 作为首批正式真实批次；`v1/v2` 仅是过程性失败尝试，不作为正式导入里程碑口径
- 当前把 `2026-04-13-radishflow-suggest-edits-poc-real-v4` 作为第二批正式真实批次；扩样本面时优先重放已有 raw dump 并按当前 runtime 重新归一化，而不是一遇到 task-level 漂移就先去折腾 provider 配置
