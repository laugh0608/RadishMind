## 关键决策

- 平台配置摘要以存在性、状态、字段来源和必填字段为主，不把 secret 或真实上游 URL 原文写入 summary。
- `mock` provider 不要求 credential；`Ollama` 视为本地 daemon，credential 可选；其它远端 provider 默认要求 `model`、`base_url` 与 credential。
- 配置文件先采用 JSON，避免为了本地 deployment smoke 过早给 `Go` 平台层引入 YAML 解析依赖。
- 本地 wrapper 只收口工作目录、`GOCACHE`、默认本地配置文件发现和 `serve/config-summary/config-check` 命令，不引入后台进程管理或 system service 语义。
- 配置检查先作为本地开发和快速门禁入口，不等同于 production secret management；正式 secret 管理和部署环境隔离仍留到后续 deployment 主线。
- 结构化 diagnostics 只做启动前 failure boundary：配置必填字段、bridge provider registry 和 inventory 可用性是本阶段稳定门禁；外部 provider 真实健康探测、进程守护和 secret backend 不在本次范围。
- provider/profile discoverability 的真相源仍是 Python provider inventory；`Go` 层只把它翻译为 northbound model catalog、selection metadata 和 diagnostics 摘要，不复制第二套 provider 配置解析。
- 请求级观测只覆盖平台排障所需的最小字段，不扩成完整监控平台；错误响应输出稳定 code 与 failure boundary，不输出真实 provider URL 或 credential 原文。
- `P1 Runtime Foundation` 在请求级观测和错误分类补齐后达到 short close，后续默认切到 `P2 Session & Tooling Foundation`，不继续横向扩同层 provider/config/diagnostics/observability 细节。
- `check-repo --fast` 可以检查 committed 事实、契约和记录摘要，但不应默认要求开发者拥有某次本地模型运行的 `tmp/` 产物；真实产物审计保留为显式 local artifact mode。
- `SessionRecord` v1 只固定 `session_id / turn_id`、bounded history policy、recovery record 和 audit 边界，不代表已经落地长期记忆、持久 session store 或自动恢复执行器。
- `Tooling Framework` v1 只固定 tool definition、registry policy、timeout/retry 和 audit record；registry 默认 `execution_enabled=false`、`network_default=disabled`、`durable_memory_enabled=false`，candidate builder 仍必须走上层确认。

## 风险与未完成项

- 当前 discoverability 已对齐 `/v1/models`、请求选择和 diagnostics；外部 provider 健康探测、部署环境分层、process supervision 和 production secret backend 仍不在本阶段范围。
- 当前 tooling 只有契约、fixture 和 audit smoke，还没有真实 executor、tool result cache、durable tool store 或跨轮恢复执行。
