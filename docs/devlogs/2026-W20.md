## 关键决策

- 平台配置摘要以存在性、状态、字段来源和必填字段为主，不把 secret 或真实上游 URL 原文写入 summary。
- `mock` provider 不要求 credential；`Ollama` 视为本地 daemon，credential 可选；其它远端 provider 默认要求 `model`、`base_url` 与 credential。
- 配置文件先采用 JSON，避免为了本地 deployment smoke 过早给 `Go` 平台层引入 YAML 解析依赖。
- 本地 wrapper 只收口工作目录、`GOCACHE`、默认本地配置文件发现和 `serve/config-summary/config-check` 命令，不引入后台进程管理或 system service 语义。
- 配置检查先作为本地开发和快速门禁入口，不等同于 production secret management；正式 secret 管理和部署环境隔离仍留到后续 deployment 主线。
- 结构化 diagnostics 只做启动前 failure boundary：配置必填字段、bridge provider registry 和 inventory 可用性是本阶段稳定门禁；外部 provider 真实健康探测、进程守护和 secret backend 不在本次范围。
- provider/profile discoverability 的真相源仍是 Python provider inventory；`Go` 层只把它翻译为 northbound model catalog、selection metadata 和 diagnostics 摘要，不复制第二套 provider 配置解析。
- 请求级观测只覆盖平台排障所需的最小字段，不扩成完整监控平台；错误响应输出稳定 code 与 failure boundary，不输出真实 provider URL 或 credential 原文。
- `P1 Runtime Foundation` 在请求级观测和错误分类补齐后达到 short close，后续默认切到 `P2 Session & Tooling Foundation`，不继续横向扩同层 provider/config/diagnostics/observability 细节。
- `check-repo --fast` 可以检查 committed 事实、契约和记录摘要，但不应默认要求开发者拥有某次本地模型运行的 `tmp/` 产物；真实产物审计保留为显式 local artifact mode。
- `SessionRecord` v1 只固定 `session_id / turn_id`、bounded history policy、recovery record 和 audit 边界，不代表已经落地长期记忆、持久 session store 或自动恢复执行器。
- `Tooling Framework` v1 只固定 tool definition、registry policy、timeout/retry 和 audit record；registry 默认 `execution_enabled=false`、`network_default=disabled`、`durable_memory_enabled=false`，candidate builder 仍必须走上层确认。
- tool result cache v1 只允许 metadata-only 或受限引用进入 session recovery checkpoint；没有真实 executor 前，不得声明 materialized result，也不得把 cache 写成长期记忆。
- recovery checkpoint v1 是审计和恢复读取边界，不是执行计划；即使 `replayable=true`，也必须保持 `auto_replay_enabled=false` 且高风险动作需要人工确认。
- checkpoint read route smoke 只允许 metadata-only fixture 边界；即使暴露 HTTP route，也不得声明 durable checkpoint store、materialized result reader 或 replay executor 已实现。
- checkpoint read route 必须把 tool audit 治理摘要保持在 metadata 层，并显式拒绝 materialized result、result ref、executor ref、durable memory 与 replay 类请求；这仍不代表真实 executor、长期记忆或 replay 能力已经存在。
- P2 promotion gate 当前只能声明 contract 与 metadata smoke 稳定；真实 executor、durable store、result materialization、长期记忆和 automatic replay 必须等待上层确认流、executor 边界、storage backend、独立审计记录和负向回归明确后再启用。
- P2 负向 fixture 必须有明确消费者；新增负向边界时，不能只新增 fixture 或文档，还要同步消费 summary 和对应 check。
- Checkpoint read route smoke 只能声明当前 Go route 已覆盖的 fixture-backed metadata-only 边界；不得把 route smoke 写成真实 checkpoint API、storage backend 或 replay executor。
- P2 readiness 当前只能声明 contract 与 metadata smoke readiness；executor、durable storage 和上层 confirmation flow 仍必须作为启用前置条件独立设计、独立审计和独立负向回归后再推进。
- P2 implementation preconditions 必须先保持 `not_ready` 可检查；后续若要解锁任一类前置条件，应先更新任务卡、fixture、readiness summary 和负向回归，而不是直接实现 executor、store 或 replay。
- P2 negative regression skeleton 只固定失败边界、错误码和 forbidden outputs；它不是完整 `negative_regression_suite`，不得据此解除 executor、storage 或 confirmation 的 `not_ready` 状态。
- P2 close candidate 只代表治理门禁齐备度已能被统一盘点；在上层确认流、executor 边界、storage backend、result materialization policy、独立 audit record 和完整负向回归满足前，不进入 P2 short close。
- P2 independent audit records design 只把审计记录形状、event source 和 confirmation / executor / storage 职责分离收口为可检查边界；它不是 durable audit store，也不解除真实 executor、storage、confirmation 或负向回归前置条件。
- P2 result materialization policy design 只把 metadata-only、result ref 和 materialized result 的当前拒绝边界收口为可检查口径；它不是 materialized result reader 或 durable result store，也不解除 executor、storage、confirmation 或负向回归前置条件。
- P2 executor boundary design 只把未来 executor 的 sandbox、allowlist、envelope、timeout / retry 和失败边界收口为可检查口径；它不是 real_tool_executor，也不解除 storage、confirmation 或负向回归前置条件。
- P2 storage backend design 只把 session / checkpoint / audit / result store 的职责分离、写入边界和 retention / redaction / secret 前置项收口为可检查口径；它不是 durable store，也不解除 confirmation 或负向回归前置条件。
