# 2026-W20

## 本周目标

- 继续推进 `P1 Runtime Foundation`，优先把 `Runtime Service` 的 provider/profile discoverability、配置分层和部署壳收口成可验证入口。

## 完成情况

- 已新增 `services/platform/internal/config.Config.SanitizedSummary()`，把平台启动配置脱敏收口为稳定 summary：`provider`、`profile`、`model`、`model_configured`、`base_url_configured`、`credential_state`、timeout、listen addr、Python bridge 和缺失必填字段。
- 已为 `radishmind-platform` 新增一次性 `config-summary` 与 `config-check` 子命令；它们复用 `config.LoadFromEnv`，不启动长驻 HTTP 服务，也不输出 `RADISHMIND_PLATFORM_API_KEY` 或 `base_url` 原文。
- 已新增 `scripts/check-platform-config.py` 并接入 `check-repo --fast`，覆盖默认 `mock`、远端 provider 配置成功、远端 provider 缺失 `base_url/credential` 失败，以及 `Ollama` credential optional 的配置状态。
- 已同步 `services/platform/README.md` 与 `scripts/check-platform-runbook.py`，把配置摘要命令、脱敏字段和 runbook drift 检查纳入平台本地启动说明。

## 关键决策

- 平台配置摘要以存在性、状态和必填字段为主，不把 secret 或真实上游 URL 原文写入 summary。
- `mock` provider 不要求 credential；`Ollama` 视为本地 daemon，credential 可选；其它远端 provider 默认要求 `model`、`base_url` 与 credential。
- 配置检查先作为本地开发和快速门禁入口，不等同于 production secret management；正式配置文件、secret 管理和部署环境隔离仍留到后续 deployment 主线。

## 风险与未完成项

- 当前配置来源仍主要是环境变量，还没有正式配置文件、环境层级覆盖规则或 secret backend。
- `config-summary` 只描述平台进程级配置，不替代 Python provider inventory 中的多 profile discovery。

## 下周建议

- 继续补长驻部署壳和配置文件层级，明确 env / file / profile 的优先级与审计输出。
- 在配置层级稳定后，把 service ops smoke 扩展到更接近 deployment 的启动包装和 failure boundary。
