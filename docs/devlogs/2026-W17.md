## 关键决策

- `RadishFlow` clean formal batch 的 negative replay 不从 failed audit entries 推导，而是要求显式 committed negative sample 反链到真实 record，避免把正向 clean 批次本身误判成失败来源
- 负例样本允许同时保留真实 `candidate_response_record` 与内联变异后的 `candidate_response`；record 用于追溯与对齐校验，内联 response 用于实际负例回归
- `suggest_flowsheet_edits` 的 replay / real-derived 治理顺序固定为“same-sample 顺序坏法 -> cross-sample record 错配 -> real-derived committed pattern”，避免直接跳到 derived 而缺 replay 对齐层
- `suggest_flowsheet_edits` 的首批 real-derived pattern 先围绕已存在 clean formal batch 里的稳定顺序漂移收口，不额外引入新的任务语义变量
- `suggest_ghost_completion` 先按“same-sample 边界漂移 -> cross-sample 记录错配”两步收口 replay；cross-sample 第一批优先选择可组成重复分组的稳定错配，而不是追求表面覆盖面
