## 关键决策

- `RadishFlow` clean formal batch 的 negative replay 不从 failed audit entries 推导，而是要求显式 committed negative sample 反链到真实 record，避免把正向 clean 批次本身误判成失败来源
- 负例样本允许同时保留真实 `candidate_response_record` 与内联变异后的 `candidate_response`；record 用于追溯与对齐校验，内联 response 用于实际负例回归
- `suggest_flowsheet_edits` 的 replay / real-derived 治理顺序固定为“same-sample 顺序坏法 -> cross-sample record 错配 -> real-derived committed pattern”，避免直接跳到 derived 而缺 replay 对齐层
- `suggest_flowsheet_edits` 的首批 real-derived pattern 先围绕已存在 clean formal batch 里的稳定顺序漂移收口，不额外引入新的任务语义变量
- `suggest_ghost_completion` 先按“same-sample 边界漂移 -> cross-sample 记录错配”两步收口 replay；cross-sample 第一批优先选择可组成重复分组的稳定错配，而不是追求表面覆盖面
- `M3` 的正式治理报表不再只汇总“是否接通”，还要稳定输出标准化覆盖摘要和优先级队列；这样主线判断可以直接基于仓库内机读报告，而不是继续依赖周志人工串联
- 当 `teacher_comparison_candidates` 已阶段性清空后，`suggest_flowsheet_edits` 的下一步不再写成抽象的“继续扩高价值真实样本池”，而要显式收口成脚本可调用的 sample group，避免每轮 capture 又回到人工临时挑样本
- `suggest_ghost_completion` 的下一步也不再只写成抽象的“继续扩高价值真实样本池”，而应优先按共享 sample-group 入口推进；治理报表若已能明确指出 `next_real_capture_group`，后续就不应再回到手工挑样本
- 对 `suggest_flowsheet_edits` 的高价值真实批次，若失败已收口为 runtime canonicalization 漂移，应优先沿现有 dump 做 `recanonicalize-response` 重导，而不是直接重打一整批 provider capture
- 对 `suggest_flowsheet_edits` 的 cross-object reconnect action，若前序已识别 connected-unit contextual warning，则最终 action citation 生成也必须继续保留这组 contextual diagnostic index，不能在 merge 阶段把它们丢掉
