## 关键决策

- `dev` 作为常态开发分支，`master` 只接受 PR
- 当前阶段只保护 `master`，不强制保护 `dev`
- PR required checks 当前收口为 `Repo Hygiene` 与 `Planning Baseline`
- 当前仓库级验证入口统一为 `pwsh ./scripts/check-repo.ps1` 与 `./scripts/check-repo.sh`
- 当前基线不照搬 `RadishFlow` 的 `Rust Baseline`，而改为适配规划期仓库的 `Planning Baseline`
- `RadishMind` 当前正式定位为两个项目的外部智能层，而不是任一业务系统的真相源
- `RadishFlow` 第一批任务优先围绕 `FlowsheetDocument`、选择集、诊断摘要、求解状态和控制面状态展开，不直接侵入求解热路径
- `Radish` 第一批任务优先围绕固定文档、在线文档、论坛内容、Console 权限知识和附件协议展开，不先走自动治理或自动写回
- 统一协议当前正式采用“通用骨架 + 项目专属上下文块”，不把两个项目强行压成同一业务字段集合
- 统一协议当前先以 `JSON Schema` 作为仓库内正式契约形式，后续再决定是否同步生成其他语言类型
- `RadishFlow` 首批任务当前正式冻结为三张任务卡，再进入样本补齐与适配器骨架阶段
- `minimind-v` 当前正式定位为 student/base 候选与实验底座，不预设为第一阶段主模型路线
- `Release Checks` 当前正式使用独立 job 名，避免远端 ruleset 把 release job 和 PR job 混成同一个 required check

- `RadishFlow` 当前 `rf-canvas` 和真实画布截图资产仍未成熟，后续 VLM 路线需要等待更稳定的 UI / 画布上下文
