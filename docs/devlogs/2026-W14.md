## 关键决策

- `dev` 作为常态开发分支，`master` 只接受 PR
- 当前阶段只保护 `master`，不强制保护 `dev`
- PR required checks 当前收口为 `Repo Hygiene` 与 `Planning Baseline`
- 当前仓库级验证入口统一为 `pwsh ./scripts/check-repo.ps1` 与 `./scripts/check-repo.sh`
- 当前基线不照搬 `RadishFlow` 的 `Rust Baseline`，而改为适配规划期仓库的 `Planning Baseline`
- `RadishMind` 当前正式定位为两个项目的外部智能层，而不是任一业务系统的真相源
- `RadishFlow` 第一批任务优先围绕 `FlowsheetDocument`、选择集、诊断摘要、求解状态和控制面状态展开，不直接侵入求解热路径
- `Radish` 第一批任务优先围绕固定文档、在线文档、论坛内容、Console 权限知识和附件协议展开，不先走自动治理或自动写回
- `Radish` 当前先只冻结 `answer_docs_question` 这一最小场景，而不是一次性平铺全部首批任务卡
- 统一协议当前正式采用“通用骨架 + 项目专属上下文块”，不把两个项目强行压成同一业务字段集合
- 统一协议当前先以 `JSON Schema` 作为仓库内正式契约形式，后续再决定是否同步生成其他语言类型
- `RadishFlow` 首批任务当前正式冻结为三张任务卡，再进入样本补齐与适配器骨架阶段
- `minimind-v` 当前正式定位为 student/base 候选与实验底座，不预设为第一阶段主模型路线
- `Release Checks` 当前正式使用独立 job 名，避免远端 ruleset 把 release job 和 PR job 混成同一个 required check
- `Radish` 文档问答当前先把召回输入约束收口为 `retrieval_expectations + artifact metadata`，再考虑真实检索器或模型输出接入
- `check-radish-docs-qa-eval.*` 当前作为仓库入口保留，真正的 `answer_docs_question` 回归逻辑统一放在 `run-radish-docs-qa-regression.*`
- `RadishFlow` 当前先围绕 `explain_diagnostics` 建立最小回归闭环，再扩展到 `suggest_flowsheet_edits` 与 `explain_control_plane_state`

## 风险与未完成项

- GitHub ruleset 仍需在仓库后台导入或更新，仓库文件本身不能自动强制生效
- 当前 `Planning Baseline` 主要覆盖治理和文档，不代表后续实现期的完整技术基线
- `RadishFlow` 当前 `rf-canvas` 和真实画布截图资产仍未成熟，后续 VLM 路线需要等待更稳定的 UI / 画布上下文
- `RadishFlow explain_diagnostics` 虽已补首批样本，但样本规模仍小，尚未覆盖全局诊断、复杂多对象关联和控制面交叉状态
- `Radish` 虽然已有清晰的文档、Console 和社区主线，但 `RadishMind` 第一批落点到底是文档应用、论坛创作器、Console 助手还是独立入口，仍待后续收口
- `Radish` 虽已先落 `answer_docs_question` 任务卡，但首个真实入口、知识源装配方式和真实检索器接口仍待继续收口
- `Radish` 文档问答的离线回归已能校验召回输入边界与 `golden_response` 输出对照，但真实召回实现和候选模型输出接入仍未落地
- 统一协议虽已落为真实 schema，但当前仍只冻结通用骨架与最小项目上下文字段，尚未进入适配器代码与更严格任务级校验
- 数据集目录与样本格式已建立，`Radish` 文档问答样本也已有首批 golden response，但更大规模样本维护流程与真实候选输出回灌仍需进一步细化
