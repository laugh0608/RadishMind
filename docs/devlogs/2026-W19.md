## 关键决策

- 当前项目今天最该推进的不是继续扩 `suggest_flowsheet_edits` 或 `suggest_ghost_completion` 的真实批次，而是先回答结构化输出约束是否足以改变路线判断
- `--repair-hard-fields` 继续保留为显式后处理工程轨，但不再作为 raw 晋级、训练准入或模型扩尺寸的直接证据
- 在更强输出约束手段尚未比较前，不默认把下一步写成 `3B/4B` 扩尺寸，也不默认继续延长同一样本 timeout
- raw -> repaired 的同样本同 timeout 差异已经足够说明：当前不能把 v2 失败直接归因为模型容量不足；应优先验证结构化输出约束和 response builder / tooling 分工
- `--inject-hard-fields` 被确认有用但不足：它能治理顶层硬字段、部分 action patch 和 issue schema minimum，但不能单独恢复非冻结 answer/citation，也不能解决复杂 cross-object prompt 的本地 CPU timeout 边界
- `suggest_flowsheet_edits` 的分工实验已被真实本地 v2 轨验证：模型负责意图/解释文本，builder 负责 action、citation、risk 与 confirmation 结构，可以消除该任务在 injected 轨中的剩余阻塞
- builder 轨的 overall blocked 说明当前问题不再是单一 `suggest_flowsheet_edits` 结构组装，而是需要可组合的 task-scoped 结构化输出策略，覆盖 ghost action patch 与 docs evidence-gap status/risk 边界
- task-scoped builder 先作为路线决策实验轨，不升级为 production contract，也不替代 raw 能力晋级证据；下一步真实本地轨若通过，路线应优先收口到 response builder / tooling 分工，而不是直接切 `3B/4B`
- 真实 task-scoped builder 轨已通过机器指标，因此当前路线应优先推进 task-scoped response builder / tooling 分工；但自然语言 merge/fallback 质量需要单独设门禁，不能被 schema/task/offline eval 通过掩盖
- 自然语言 guardrail 是 builder/tooling 路线的质量门禁，不是 raw 能力提升证据；修正后同一 v2 本地轨已证明目标坏文本可以被 fallback 替代，但后续扩大样本面前仍需要系统化 review / audit，而不能只依赖路径合并统计
- 自然语言 audit 已接入 `check-repo`，因此后续扩大 task-scoped builder 样本面时必须同时维护 fallback 使用比例与已知坏文本回归门禁；该门禁仍是 deterministic tooling 检查，不是 raw 模型晋级证据
