## 关键决策

- `v92` 之后的下一轮真实 capture 先聚焦 action filtering 与 citation/action 交错边界，而不是继续增加同质化 replay 或重复已完成的八组高价值入口
- 新入口仍复用现有 33 条离线样本中的高价值组合，先验证组合压力面是否继续暴露 runtime canonicalization 漂移；只有需要全新语义时，再新增离线样本本体
- `DOWNSTREAM_SEPARATOR_STATE_DEPENDENT` 这类只有 dependent warning 且已有 error stream diagnostic 支撑的既有 action，不应因 target 是 stream 就绕过 warning-only action 过滤
- 当前真实样本扩展降级为条件触发：必须先有新的非重复高价值 drift 假设，且优先来自服务/API 接入或集成演示暴露出的评测缺口
- `M3` 后半段的开发主线正式从“样本数量推进”转向“服务/API 边界推进”：先固定 Gateway 的路由、错误、metadata 和回归门禁，再评估是否需要新的真实 capture 补洞
- `RadishFlow` 集成演示切片优先复用 committed export/eval fixture 和 mock provider；只有该链路暴露现有 fixture 无法覆盖的新服务/API 风险时，才考虑新增样本或真实 capture
- `CopilotGatewayEnvelope` 的 `metadata` 是调用侧日志与审计信号，不是业务真相源；`response.proposed_actions` 即便在 `partial` 响应中存在，也仍必须保持 advisory-only 和人工确认边界
- `services/gateway` 的最小职责已从“待定义”推进到“已有纯 Python 骨架和仓库级 smoke”；当前未决点不再是是否需要 gateway，而是首个对外调用形态优先进程内 API、HTTP JSON 服务，还是两者并行
- `RadishMind-Core` 属于本项目的自研主模型，但“自研”默认指开源基座上的领域数据、任务协议、风险约束和评测偏好适配，不指从零预训练基础大模型
- 图片输入理解可以进入 `RadishMind-Core` 或其视觉适配路线；图片像素生成应保持独立 backend 能力，由主模型输出结构化意图、约束和审查信息后再调度
- 在 `32GB` 级本地开发 / 小型部署设备上，`14B` / `32B` 不作为 `RadishMind-Core` 的默认自研目标；若未来要上更大模型，必须先升级部署硬件、训练数据和评测基线
- `M3` 的首个 Gateway 调用形态优先进程内 Python API；当前不启动长驻 HTTP 服务，不把鉴权、端口、部署生命周期混入这轮最小切片
- `gateway-service-smoke-summary` 与 `radishflow-gateway-demo-summary` 分别固定通用 envelope 行为和 `RadishFlow export -> adapter -> gateway` 集成行为，后续 gateway metadata、错误 envelope 或确认语义变更必须同步更新 summary 或解释兼容性
- `radishflow-gateway-ui-consumption-summary` 固定的是调用侧视图模型语义，不是完整 UI 实现；它只证明上层能安全展示和审计候选提案，不能替代真实 `RadishFlow` UI 接入或命令层确认流
- `radishflow-candidate-edit-handoff-summary` 固定的是命令层交接语义，不是命令执行；即使用户确认，当前本仓库也只生成 proposal handoff，不直接修改 `FlowsheetDocument`
- 在上层项目尚未准备接入模型 / Agent 前，`RadishMind` 不把真实上层调用位置或 HTTP JSON 包装作为阻塞项；当前应先把 `RadishMind-Core` 基座评估、样本格式和 image adapter schema 做实
- `CopilotTrainingSample` 只承载训练 / 蒸馏数据流转边界，不替代 `CopilotRequest` / `CopilotResponse`；嵌套请求和响应仍必须分别通过既有正式 schema
- `suggest_ghost_completion` 的 `manual_only` 中风险候选不等同于高风险写回动作；训练样本安全校验应保持 `advisory`，并只对目标响应要求确认、`high` 风险动作、非 `manual_only` 的中风险执行边界动作强制 `requires_confirmation_for_actions=true`
- `teacher_capture` 训练样本必须来自 audit pass 的 committed candidate record；audit failed、schema-invalid、citation/risk 边界不稳定或未被转换 manifest 显式列入的 record 不得进入训练集合
- 训练样本 JSONL 当前默认是本地生成产物；仓库 committed 资产优先提交 manifest、summary、复核记录和实验说明，只有小型 fixture 有明确用途、来源、样本数、复核状态和退场条件时才考虑入仓
- 当前 seed set 默认全量复核；后续更大 candidate record 池进入训练集合前，按 `project / task / source / provider / model / risk_level / requires_confirmation / coverage_tags / batch_id` 分层抽样，默认每层至少复核 `20%` 且不少于 `5` 条，高风险、确认边界、新任务、新 provider/model、新 action/patch 结构、新来源族、schema 版本变化和历史失败模式必须全量复核
- 更大训练集合必须保留离线评测 holdout，默认至少 `10%` 且每个任务至少 `3` 条；holdout 不得进入同一训练 JSONL split，避免训练 / 评测泄漏
- 当前 holdout split 只采用未列入现有 golden_response / teacher_capture 转换 manifest 的 committed eval 样本；在真实训练 split 生成前，必须继续用检查脚本防止 overlap
- `RadishMind-Core` 首轮按 `3B` 优先、`4B` 对照、`7B` 延后评估；`14B/32B` 不进入默认目标，图片像素生成继续由 image adapter / backend 承接
- `RadishMind-Core` 首轮离线评测的阻塞指标先固定为 schema 合法率、citation 对齐率、风险确认保持率、高风险动作确认率、advisory 边界保持率和 docs QA 检索来源约束；当前只固定阈值，不运行模型
- `RadishMind-Core` 离线评测记录在 planned 阶段不得填入 observed metric 或 passed 结论；只有明确执行离线模型评测后，才允许补真实观测值、成本观测和证据路径
- `RadishMind-Image Adapter` 的第一版 schema 链路只固定结构化生成意图、backend request 与 artifact metadata，不承诺具体 backend 常驻、训练方式或像素生成质量
- `safety.requires_confirmation=true` 的 image generation intent 不得被视为可直接提交生图 backend；当前由 `check-image-generation-intent-contract.py` 同时在 intent 与 backend request 层作为仓库级回归边界固定
- `RadishMind-Image Adapter` 的首个评测 manifest 只评估可审计的结构化链路和安全门禁，不评价图片像素质量，也不允许 provider raw dump、图片像素、权重、checkpoint 或大规模 JSONL 作为 committed 资产
- `RadishMind-Core` 离线评测 runner 的 fixture-run 即使所有指标为 `1.0`，也不得自动代表真实模型晋级；当前 `promotion_status` 仍保持 `no_promotion_planned`，下一步应接真实 candidate output 再做人工复核
- `RadishMind-Core` candidate wrapper 的 `golden_fixture` dry-run 只证明本地推理前后的工程链路可复跑，不代表模型已经产出第一条真实对话；第一条真实模型 candidate response 必须来自用户明确提供或批准下载后的本地模型目录
- `Qwen2.5-0.5B-Instruct` 已能在当前 WSL CPU 环境产出第一条本地回答，但当前结果只是 schema-invalid 观测，不进入离线评测通过记录；后续应先收紧 prompt 包装和 JSON schema 输出策略，再评估是否下载 `1.5B` 或更大模型
- v8 观测后不再把“继续加长 0.5B prompt”作为默认主线；后续优先做更高容量本地模型对照、解码后结构化修复策略或 schema-constrained 输出能力验证，并继续保留当前 wrapper 的完整失败记录能力
- `Qwen2.5-1.5B-Instruct` 在同批 fixture 上能提高 schema 合法率并拿到首条 task-valid ghost completion，但仍不能稳定保持 `CopilotResponse` 任务级硬字段；因此当前不应把 1.5B 直接视为可晋级 student 基座，应先验证 schema-constrained decoding、后处理修复和更小 fixture 子集的稳定性
- `--repair-hard-fields` 只能作为显式后处理实验和工程可行性验证，不能替代 raw 模型能力评估；后续任何晋级判断都必须同时保留 raw summary、repaired summary、修复路径统计和人工复核结论
- 本地候选生成必须具备 per-sample timeout 边界；timeout 是可审计失败，不是 schema 通过或模型质量通过，应继续通过 `--allow-invalid-output` 和离线评测记录进入阻塞指标
- 当前 1.5B 全量 raw / repaired 对照必须使用同一 timeout 档位；默认 `240s` 是评测治理基线，不因 repaired 后处理另给更长生成预算，避免把生成预算差异误读成模型能力或后处理收益
- `240s` 档位已通过当前环境复跑确认，但不是跨硬件、跨模型尺寸或跨样本面的永久常量；repaired 复跑最慢样本已接近边界，后续必须把 `max_generation_seconds`、`timeout_count` 与 `hit_max_new_tokens_count` 作为每轮本地候选摘要的必读字段

- JSON 尾逗号清理只作为本地候选 wrapper 的解析容错和可审计指标，不应被解读为模型已经严格遵循 JSON；summary 必须继续记录 `json_cleanup_applied`，并保留 raw/repaired 区分
