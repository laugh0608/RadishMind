## 功能目标

让内部开发者从用户工作区的应用列表选择一个应用后，在同一应用详情中完成模型发现、接入示例生成、测试调用和调用审查。该路径消费现有 `/v1/models`、Gateway 调试台与脱敏请求历史，不新增上行路由、协议适配器、SSE 解析器、请求状态模型、存储库或模型服务注册表。

本功能仍属于内部开发者预览。独立专题已经提供开发测试态 API 密钥生命周期，但本功能的调用成功只证明当前本地 Gateway 路径可交互、可取消和可审查，不代表生产 API 分发、真实模型服务 SLA、生产认证、生产密钥生命周期、配额或计费已经成立。

## 目标用户与连续流程

目标用户是需要把某个工作区应用接到 RadishMind Gateway，并在提交真实集成改动前验证协议和审查调用记录的内部开发者。

1. 用户从应用列表选择一个应用，并进入现有应用详情。
2. 应用 API 接入工作区显示当前工作区 / 应用作用域；切换应用会立即清空旧模型、旧示例选择、旧失败和旧调用交接状态。
3. 默认离线模式只显示能力说明并保持零网络请求。显式开发测试态模式下，用户使用开发身份头，或从 API 密钥页把一次性 Bearer 凭据交给 Playground 后，主动加载或刷新 `/v1/models`。
4. 模型目录在 `loading`、`ready`、`empty` 或 `failed` 之间转换；只有通过响应校验的模型标识才能进入选择器。
5. 用户选择模型、Chat Completions / Responses / Messages 协议和 cURL / Python / TypeScript 语言，生成只含环境变量占位符的接入示例。
6. 用户将当前应用、协议和模型交给现有 Gateway 调试台；调试台继续负责单次响应、流式响应、取消、稳定失败和当前内存态输入输出。
7. 调用完成后，调试台用同一 `request_id`、应用作用域和实际 `consumer_ref` 交给现有请求历史，并精确打开脱敏详情。

## 应用作用域

- 接入工作区的应用只能来自当前应用列表选择，不能使用启动器中的固定应用作为当前业务选择。
- `dev_headers` 模式下，`/v1/models` 请求携带当前租户、工作区、消费方、应用、主体、作用域和请求标识调用方请求头；`api_key_dev_test` 模式只发送 Bearer 凭据，并由服务端从密钥记录恢复可信作用域。两类凭据互斥，均不进入界面、示例或持久化介质。
- 调试台交接详情固定包含 `applicationId`、`protocol` 和 `model`，不包含输入、输出、凭据或完整请求头。
- API 密钥页到调试台的一次性交接是独立内存事件；API 接入区只接收经过严格校验的模型公开字段，不接收或转发 Bearer 凭据。
- 调试台发出的上行请求必须使用交接应用覆盖静态默认值；请求历史交接必须继续使用同一应用作用域和实际消费方，否则详情读取按失败关闭处理。
- 切换应用时，正在加载的目录请求应被取消或结果被丢弃；新应用不得显示旧应用的模型、示例选择、失败或请求标识。

## 模型目录状态与响应校验

模型目录状态固定为：

- `offline`：未启用显式开发测试态数据源；不发起请求。
- `idle`：已启用，但用户尚未加载当前应用的模型。
- `loading`：当前应用的 `/v1/models` 请求进行中；允许显式取消或被应用切换淘汰。
- `ready`：响应通过校验且至少包含一个模型。
- `empty`：响应通过校验但 `data` 为空。
- `failed`：HTTP、网络、JSON、schema、关联标识或禁止字段投影校验失败。

响应只接受 `object: "list"`、`data: Model[]`。每个模型必须包含单行、非空且长度受限的 `id`，并校验 `object: "model"`、非负 `created` 与字符串 `owned_by`。界面只保留模型标识、所有者和公开协议能力；忽略模型服务凭据状态、端点与未知元数据。顶层或模型公共投影出现凭据、授权、敏感信息、原始错误、端点或请求头载荷时按失败关闭处理。

HTTP 非成功响应保留 `gateway_model_catalog_http_failed`，非法 JSON / 信封保留 `gateway_model_catalog_response_invalid`，网络失败保留 `gateway_model_catalog_network_error`；界面只显示稳定错误码与脱敏摘要，不显示响应正文、调用栈、端点或请求头。

## 接入示例

| 协议 | 路由 | 请求结构 |
| --- | --- | --- |
| Chat Completions | `/v1/chat/completions` | `model + messages[] + stream` |
| Responses | `/v1/responses` | `model + input + stream` |
| Messages | `/v1/messages` | `model + messages[] + max_tokens + stream` |

每种协议提供 cURL、Python 和 TypeScript 示例，并遵守以下约束：

- 基础 URL 与密钥只使用 `${RADISHMIND_BASE_URL}`、`${RADISHMIND_API_KEY}` 或对应进程环境变量引用。
- 模型使用当前经过目录校验的标识；临时输入使用 `${RADISHMIND_INPUT}` 或运行时变量，不嵌入调试台当前输入。
- 不显示真实密钥、密钥哈希、`Authorization` 值、cookie、凭据引用、模型服务端点或任何 `X-RadishMind-Dev-*` 调用方请求头。
- 示例不是凭据签发或生产认证文档；环境变量缺失时由示例运行环境显式失败，不提供默认密钥。
- 示例通过纯函数生成，只存在当前组件内存；v1 不提供保存、分享、导出或生成应用配置文件。

## 调试台与请求历史交接

- 接入工作区只派发应用 / 协议 / 模型交接，并导航到现有调试台。
- 调试台接收交接后重置旧输入、输出和结果，保留默认临时输入模板；用户显式选择单次响应或流式响应并发送。
- 调试台继续使用既有请求标识生成、请求构建器、响应适配器、SSE 终态解析器、输出预算、取消和稳定失败映射。
- 成功、失败或取消后，请求历史交接派发 `requestId`、`applicationId` 与当前 `consumerRef`。请求历史清空旧筛选，以对应作用域读取列表和详情，并精确展示同一请求标识。
- 请求历史不可用或作用域不匹配时显示交接失败；调试台当前结果仍留在内存，不伪造已持久化或已审查状态。

## 隐私与持久化边界

- 默认离线模式必须保持零请求；离线模型目录为空且不回退到已提交 fixture。
- 模型选择、协议、语言、临时输入、当前输出、流式增量、失败、请求标识以及 API 密钥模式下的 Bearer 凭据只存在于对应 React 组件内存；API 接入区只消费脱敏模型目录。
- 上述值不得写入 URL 查询 / 锚点载荷、`localStorage`、`sessionStorage`、草案、工作流运行、Gateway 请求历史载荷、产物、日志、周志或业务真相源。
- URL 只使用稳定区段锚点，不包含应用、模型、协议、请求标识、输入或输出。
- Gateway 请求历史继续只保存既有脱敏运行元数据；本功能不改变其 schema、保留策略或存储库。

## 实施拆分

1. 建立独立延迟加载接入工作区面板与纯消费端：配置、模型目录校验、示例生成和应用状态重置。
2. 扩展现有浏览器内调试台交接事件，使面板可接收应用 / 协议 / 模型；不复制调用逻辑。
3. 让现有调试台配置在每次提交时绑定当前应用，并在交接或应用切换时清空旧状态。
4. 扩展现有请求历史交接，使详情读取使用调用对应的应用作用域；不改变后端 API。
5. 补消费端单元测试、生产构建、必要 Go 测试、真实浏览器连续流程与存储泄漏检查。

## 验收方式

单元测试至少覆盖：离线模式零请求；模型目录成功、空结果、HTTP 失败、网络失败和非法响应；`dev_headers` 与 `api_key_dev_test` 的互斥请求头；三协议 × 三语言示例的路由、请求结构与敏感信息脱敏；应用 / 协议 / 模型到调试台的交接；脱敏模型目录反向复用；请求标识 / 应用 / 消费方到请求历史的交接；应用切换后旧目录、凭据、选择、失败和请求标识不进入新状态。

真实浏览器必须在显式开发测试态配置下完成：选择应用、加载模型、切换协议与语言生成示例、至少一次单次响应、至少一次流式响应、取消请求、按同一请求标识精确打开请求历史详情，并检查控制台以及 URL、`localStorage`、`sessionStorage` 不含输入输出、密钥、应用交接载荷或请求标识。

相称验证固定为 Web 单元测试、生产构建、与作用域 / `/v1/models` 直接相关的 Go 测试、`git diff --check` 和 `./scripts/check-repo.sh --fast`。若实现中实际改变阶段边界、协议、schema 或治理真相源，再补完整 `./scripts/check-repo.sh`。
